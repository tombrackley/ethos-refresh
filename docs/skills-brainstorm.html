Discussion brief

Skills on Ethos

Skills sit at the intersection of three taxonomies — CPD regimes, organisational frameworks, and role expectations. Each names the same underlying concepts differently. If we model that once, the rest of the experience gets simple.

The core idea

Underneath everything, the platform owns one canonical graph of skills — atomic, well-defined concepts like AI governance, director duties, enterprise risk frameworks. Every external label (a CPD regime competency, an org's framework, a role expectation) maps onto those canonical skills. Users and orgs never see the canonical layer. They see their own view.

Why this is the unlock

  • Slight name differences disappear. Same concept, one entry in the user's view.
  • Activities credit multiple regimes automatically. Log a workshop once — the system fans it out to the user's CPD regime, the org's framework, and their role.
  • Adding a new regime is a mapping exercise, not a redesign.

Three surfaces, one model

Same canonical data, three filtered views — each tuned to a different question.

User
“What am I expected to know? How am I tracking?” One flat list of skills. Each row shows: required by whom (regime + org + role), current level, recent activity.
Org admin
“Where are the gaps across my team?” Same skills as a heatmap. Quickly see where capability is thin and where to invest training.
Compliance
“Are we meeting our regime obligations?” Same skills scoped to a regime. Single source of truth for audits.

Anti-complexity guardrails

The risk in this space is over-modelling. A few non-negotiables:

  • Don't show users multiple taxonomies. One display label per skill, per user. Internal mappings stay internal.
  • Don't make users self-tag activities. AI tags on log. User confirms or edits if needed.
  • Don't make proficiency levels granular. Three states is enough: Need it · Developing · Have it. Beginner→Expert is too much to track, too little signal.
  • Don't double-count progress visually. Unified progress bar. Hover or drill in for per-regime breakdown.
  • Separate “must demonstrate” from “want to learn.” Required skills are obligations. Focus areas and learning goals are aspirations. Same page; different sections; different tone.

Concepts that need a sponsor

Some Skills concepts sound tidy in the UI but have no obvious source of truth underneath. They feel like the system knows something — when really the value is asserted because we typed it into a JSON file. In a prototype this is invisible; in a real product the user immediately asks "who decided that, and how do I change it?" and there's no answer. Two we hit during this round:

Role progression

A line saying "you're 72% of the way to Partner" implies a defined matrix of role → skill → required level, the people who maintain it, and a cross-org/cross-jurisdiction agreement on what "Partner" means. None of those exist by default. The honest options:

  • Role-driven (org-defined matrices). Real but requires sustained admin work per org. Cross-org portability is hard. No org will staff this on day one.
  • Self-set. The user declares their own target role. Soft, and indistinguishable from a Learning Goal.
  • Drop it. Don't claim progress toward a role until there's a structure that defines what the role requires.

Position: drop until we have a story. The page shouldn't say "you're tracking toward X" if it doesn't actually know what X needs.

Skill targets (targetLevel)

"Target: Advanced" beside a skill implies someone (system, regulator, org) has decided what the user should be aiming for. They mostly haven't. The honest options:

  • Role / org. Same friction as role progression — requires a maintained role-skill matrix.
  • Regulator. Regulators prescribe hours and topics, not proficiency levels. "Be Proficient in AML" isn't how the rule reads.
  • Self-set. User picks a target per skill. Conceptually clean but overlaps directly with Learning Goals on the Profile.
  • Drop it. Skill has a current level, full stop. Gap-closing happens via Learn / Journeys without a numeric target.

Position: drop the target concept from Skills Profile. If we want "where do I want to get to?" framing, that's already a Learning Goal.

The pattern

If you can't name who owns the value and where it's edited, the page shouldn't render it. A tidy-looking metric without an origin story is worse than no metric — it implies the system has authority it doesn't have. When evaluating a new piece of skills data, ask: "Where does this come from, who maintains it, and how does the user change it?" If any of those three answers is hand-wave, cut it before shipping.

Setup is the biggest UX risk

Most platforms collapse at the moment you have to configure a framework. The path that stays simple:

  • Org admin picks: country, sector, primary regime(s). Platform proposes a starter skill set.
  • Role templates (NED, GC, Compliance Officer) come pre-mapped. Admin overrides per role if needed.
  • Users don't configure anything. Their skill list is derived from org + role + regime. They only pick personal focus areas and learning goals on top — the soft layer.

Four decisions worth making first

  1. Who owns the canonical graph?

    Ethos-maintained — your IP, you map new regimes (defensible moat) — vs. AI-derived per org (less precise, less maintenance). This is the single biggest call.

  2. Whose label wins on screen?

    When two regimes name the same skill differently, do users see the platform's canonical name, their primary regime's name, or their org's preferred name?

  3. One section or two?

    Required skills (org / regime — obligation framing) and Focus areas + Learning goals (personal — aspiration framing): one combined surface or two co-located but distinct sections?

  4. Who owns the progression scale?

    Three sub-questions, all about authority over the rating system itself. What does each level mean? Beginner / Intermediate / Advanced needs a shared definition — without one, the scale is opinion. Who decides when someone has moved up? Self-assessment, manager sign-off, scenario engine, third-party assessor, or some combination. How do regime competencies map onto the canonical level scale? Does completing AICD's Director Conduct course make you Intermediate or Advanced? Until these three are answered, the level shown on a skill is just an assertion the system has no real basis for.

Starting point for the conversation: Ethos owns the canonical graph; display the user's primary regime's name as the visible label (it's what they recognise); keep obligation and aspiration as two visually-distinct sections on the same Skills page.
